# Arcturus Service Blueprint

A public project template and bootstrap tool for deploying applications through the Arcturus manifest-driven, digest-pinned Quadlet control plane.

> **Status:** `v1.0.0-rc.2`, aligned with Arcturus `v1.0.0-rc.2`. Manifest v2 remains authoritative for generated projects; manifest v1 remains supported both as a generated v2-backed routing mirror and as a host-preserved native routing format during incremental migration. Gitea, GitHub, and generic CI are supported.

The blueprint adds a project-owned build graph, release template, CI adapters, lifecycle commands, acceptance probes, update provenance, and agent guidance. It does not add a second deployment engine: production ownership remains with Arcturus, rootless Podman, and user systemd.

## Ownership boundaries

| Owner | Responsibility |
| --- | --- |
| Application repository | Source, tests, Containerfiles, project build graph, release template, and non-secret runtime configuration |
| CI | Provider-neutral validation/build/publish through Gitea, GitHub, or a generic runner; external or short-lived Arcturus-owned registry credentials |
| Arcturus deployer | Authentication, schema validation, host preflight, locks, image verification, migration handoff, staging, readiness, state, and rollback |
| Quadlet/systemd | Containers, networks, volumes, timers, ordering, restart policy, boot startup, and journald |
| Registry/router | Consume the active release and publish ingress configuration and receipts |
| Terraform | Optional long-lived host/network/bootstrap infrastructure only |
| Compose | Optional local development and explicitly declared one-time migration source only |

Production releases never run Terraform, reset a host Git checkout, deploy `latest`, depend on Watchtower, or let Compose and Quadlet own the same container.

## Bootstrap a project

From a blueprint checkout, use the replayable updater wrapper:

```bash
./scripts/arcturus-update bootstrap \
  --project-dir /path/to/my-api \
  --service my-api \
  --type web \
  --image-repository registry.example.org/team/my-api \
  --domain api.example.org \
  --deploy-url 'http://<private-host-address>:9090' \
  --bundle 'registry.example.org/platform/arcturus@sha256:<64-hex-digest>' \
  --non-interactive
```

The wrapper invokes `arcturus-setup init`, installs a project-local updater, and records the normalized setup command and blueprint identity. Omit values and `--non-interactive` for guided setup:

```bash
./scripts/arcturus-update bootstrap --project-dir /path/to/project
```

Run `./scripts/arcturus-update --help` or `./scripts/arcturus-setup init --help` for complete options. `host` and `all` setup modes can also validate or invoke the Arcturus host installer.

## Generated contract

The generator creates or updates:

| File | Purpose |
| --- | --- |
| `arcturus.release.json` | Sole authoritative production release manifest template |
| `.arcturus/compat-v1.json` | Generated v2-backed manifest-v1 routing mirror; never an independent lifecycle manifest |
| `.arcturus/project.json` | Project-owned build, component mapping, CI, registry, and verification graph |
| `.arcturus/project.env` | Non-secret convenience settings and canonical replay intent |
| `.arcturus/lock.env` | Digest-pinned Arcturus tool bundle, schema lock, minimum host version, and required capabilities |
| `.arcturus/bootstrap.json` | Current blueprint provenance and normalized setup command |
| `.arcturus/bootstrap-history.jsonl` | Append-only successful bootstrap/update history |
| `scripts/arcturus-*` | Guard, CI, deploy, verify, lifecycle, acceptance, operator, and update wrappers |
| CI workflows | Gitea, GitHub, generic, or none, according to setup options |
| `AGENTS.md` | Explicit deployment contract for automation and LLM agents |

Secret values are never generated into tracked files.

Re-running setup updates generator-owned files only while they remain unchanged. Local modifications are preserved and proposed replacements are written under `.arcturus/updates/`. `--force` creates a timestamped backup before replacement. `.arcturus/project.json` is always project-owned and is never overwritten automatically.

## Update a project

For an OptiScaler-like copy-and-apply flow, replace the ignored `.arcturus/blueprint/` directory in the application repository with a newer extracted blueprint, then run:

```bash
./scripts/arcturus-update apply --dry-run
./scripts/arcturus-update apply
```

Alternatively, update directly from another checkout:

```bash
./scripts/arcturus-update apply --from /path/to/arcturus-service-blueprint
```

The updater replays `.arcturus/project.env`, preserves project-owned and locally modified files, updates itself safely, and records the blueprint version, source commit when available, tree fingerprint, and normalized command. See [Updating projects](docs/updating-projects.md).

Projects generated by the project-aware RC1 setup can be adopted using their existing `.arcturus/project.env`. Earlier webhook, Compose, or Terraform generations require a one-time explicit bootstrap because they do not contain enough durable setup intent for blind replay. See [Compatibility](docs/compatibility.md).

## Build graph

`.arcturus/project.json` maps build outputs to release components. One local image may serve multiple components such as `web` and `db-init`. External mode may push it once to a shared repository. Owned mode builds once but pushes the same image to separate `<service>/<component>` repositories so each component receives an isolated receipt. Fixed infrastructure components must already use immutable digests.

Validation targets run before release targets in isolated job-local Buildah storage. The runner does not need the application's language toolchain when tests are expressed as Containerfile validation stages.

For a declarative multi-component example, see [`examples/projects/stellar-like`](examples/projects/stellar-like). For a shared image plus one-shot asset export, see [`examples/projects/web-with-assets`](examples/projects/web-with-assets).

## Service types

- `web` — long-running HTTP service; can be public, internal, or host-bound.
- `worker` — long-running background process or queue consumer.
- `scheduled` — one-shot container invoked by a generated systemd timer.
- `oneshot` — migration, initialization, seeding, or asset-export dependency.

All types use the same release manifest, deployment API, state store, secret model, verification, and rollback path.

## Registry and CI compatibility

Use `--registry-mode external` to preserve an existing Gitea or GitHub workflow that pushes to Gitea Container Registry, GHCR, or another OCI registry. The generated workflow uses provider-correct contexts and keeps `REGISTRY_USER` and `REGISTRY_TOKEN`.

Use `--registry-mode owned --registry-origin https://<private-name>.ts.net` to let either Gitea Actions or GitHub Actions request short-lived Arcturus credentials and publish through Tailscale. No provider-specific Arcturus API exists; only the workflow context names differ. The runner must already be able to reach the private origin before `scripts/arcturus-ci` starts. Keep an existing project-owned Tailscale join step (as CrownFi does on GitHub), or use a tailnet-resident Gitea/self-hosted runner; the blueprint intentionally does not invent auth keys, tags, or ACL policy.

Manifest-v1 routing compatibility is enabled by default. The generated file is a derived compatibility export for older routing consumers, not a second deployment mechanism and not the file a current v2 host uses to activate a release. Disable it only for a project with no v1 consumer by passing `--no-compat-manifest-v1`, or by setting `ARCTURUS_COMPAT_MANIFEST_V1=false` in `.arcturus/project.env` and rerunning setup. Existing project-authored native-v1 installations remain valid on the host and should be migrated incrementally rather than overwritten.

## First deployment

1. Confirm `.arcturus/lock.env` references an immutable Arcturus RC2-or-newer bundle and records the required capabilities.
2. On the Arcturus host, create a service-scoped deployment token:

   ```bash
   umask 077
   arcturusctl token create \
     --database "$HOME/.config/arcturus/tokens.json" \
     --service my-api \
     --token-id my-api-ci \
     --output "$HOME/.config/arcturus/my-api-ci.token"
   ```

3. Store the token file's value as protected CI secret `ARCTURUS_DEPLOY_TOKEN`.
4. In external registry mode, add protected `REGISTRY_TOKEN` and optionally `REGISTRY_USER`. In owned mode, do not add long-lived registry credentials; Arcturus issues a short-lived upload credential per release.
5. Push to the protected deployment branch.
6. CI verifies host capabilities before image builds, then runs receipt-aware preflight on the rendered immutable release immediately before deployment. Preflight checks host capabilities, token scope, required Podman secrets, and external named volumes/networks.
7. CI runs `scripts/arcturus-guard`, validates build targets, pushes full-commit image tags, resolves digests, renders the release, deploys, and verifies active commit/images/health/routing.
8. Inspect runtime state from a trusted host:

```bash
export ARCTURUS_API_URL='http://127.0.0.1:9090'
export ARCTURUS_TOKEN_FILE="$HOME/.config/arcturus/my-api.token"
./scripts/arcturus-service status
systemctl --user status arcturus-my-api.target
journalctl --user -u 'arcturus-my-api-*' --since today
```

An update is another immutable release. The deployer reports success only after all required services, one-shots, timers, health checks, migration handoff, and routing verification have passed.

## Compatibility boundary

RC2-generated projects require these host capabilities:

```text
authenticated-preflight
legacy-compose-handoff
```

The generated lockfile records the minimum host version and features. `arcturusctl project preflight` rejects an incompatible host before application builds begin. The machine-readable contract is in [`COMPATIBILITY.json`](COMPATIBILITY.json).

## Lifecycle

```bash
./scripts/arcturus-service status
./scripts/arcturus-service rollback
./scripts/arcturus-service rollback --deployment-id '<known-good-id>'
./scripts/arcturus-service disable
./scripts/arcturus-service enable
./scripts/arcturus-service remove
```

`remove` withdraws generated runtime ownership and routing, but preserves release archives, audit data, Podman secrets, bind-mounted data, and named volumes.

## Acceptance

The manual CI acceptance workflow submits a deliberately unhealthy same-image release and requires failed promotion, successful automatic rollback, and restoration of the original revision, digests, and route. It does not reboot the host or operate backups.

Host operators use `scripts/arcturus-host-acceptance` for backup-gated reboot verification and must explicitly name critical units.
