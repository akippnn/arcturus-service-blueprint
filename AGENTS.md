# Arcturus deployment contract for automation and LLM agents

Read `.arcturus/project.json` and `arcturus.release.json` before changing builds, deployment, routing, lifecycle, or secrets. Manifest v2 is the sole production deployment, activation, rollback, and recovery contract.

## CI providers

Gitea Actions, GitHub Actions, and generic runners use the same provider-neutral `scripts/arcturus-ci` driver. Workflows only provide a full 40-character commit SHA and protected secrets. Use `gitea.sha` in Gitea workflows and `github.sha` in GitHub workflows. Workflow `concurrency` is advisory; the Arcturus host-side per-service lock is the authoritative serialization boundary. CrownFi and other GitHub-hosted projects may keep a project-owned Tailscale join step before `scripts/arcturus-ci`; Gitea runners may either be tailnet-resident or perform the same join. The blueprint never guesses reusable Tailscale credentials or ACL tags.

## Registry modes

- `external`: authenticate to an existing registry with `REGISTRY_USER` and `REGISTRY_TOKEN`, push full-revision tags, resolve immutable digests, and deploy manifest v2. For a Gitea package registry, use a scoped package-write PAT or deploy credential; do not assume the workflow `GITEA_TOKEN` can publish OCI packages.
- `owned`: after the runner has private network reachability, request a short-lived Arcturus upload grant, push through private Tailscale HTTPS, complete Rust verification, receive immutable receipts, and deploy the same manifest-v2 lifecycle. Only `ARCTURUS_DEPLOY_TOKEN` is long-lived in the Arcturus publisher; tailnet enrollment remains a project/runner concern.

A build may serve multiple components. In external mode a shared repository may still be used. In owned mode the image is built once, then automatically pushed to distinct `<registry>/<service>/<component>` repositories so every component receives an isolated receipt. Do not duplicate Containerfiles or rebuild the image merely to satisfy receipt ownership.

## Manifest-v1 compatibility

For projects generated by this blueprint, manifest v1 is an optional strict routing compatibility export. `scripts/arcturus-compat-v1` must generate `.arcturus/compat-v1.json` from the rendered manifest-v2 release; never author or update that generated file independently. It carries the exact v2 revision and is checked structurally against v2 for older routing consumers. A current Arcturus host routes an active v2 release directly and does not trust an authored v1 file to claim v2 provenance.

Existing native manifest-v1 installations are still supported by the Arcturus host. The registry normalizes them, strips authored provenance claims, assigns a canonical manifest digest plus content-derived revision, and the router recomputes both before accepting the route. When adopting the blueprint in such a repository, preserve the project-owned v1 source and bootstrap manifest v2 incrementally; do not replace a working native v1 file merely because the blueprint can generate a mirror.

The deprecated legacy `/deploy` endpoint is only for installations that have not yet transferred lifecycle ownership to manifest v2. It requires a service-scoped token and full immutable Git SHA. It does not provide receipt-backed artifact verification or the complete v2 rollback contract and must not be presented as equivalent to v2.

## Required workflow

1. Run `scripts/arcturus-guard`.
2. Run `scripts/arcturus-ci <full-40-character-commit>`.
3. Build validation targets before release targets in isolated job-local Buildah storage.
4. Apply local and host image-size policy before upload.
5. Render every component as `repository@sha256:digest`.
6. Deploy through `arcturusctl project deploy` and verify active revision, exact images, healthy units, and matching router receipt.
7. Use `scripts/arcturus-lifecycle` for status, rollback, enable, disable, or remove.

Every routed component joins `internal_routing`; route ports are container ports. Nginx is generated from published Arcturus state.

Forbidden: mutable tags, Watchtower, production Compose, Terraform application-release ownership, direct deployment curl, shared/global runner pruning, embedded credentials, secret-like manifest environment values, manual generated-nginx edits, or an independently maintained replacement for a blueprint-generated manifest-v1 mirror. Preserve project-owned changes and review proposals under `.arcturus/updates/`.
